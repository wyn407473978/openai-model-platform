# CLAUDE.md

请遵守本项目开发规范。本文件是 Claude Code 进入项目后的总入口，具体语言和技术栈规则见下方文档。

- Go 后端规范见：`.claude/rules/go-backend.md`
- React 前端规范见：`.claude/rules/react-frontend.md`
- PostgreSQL 规范见：`.claude/rules/postgresql.md`
- 多 Agent 协作规范见：`.claude/rules/agent-teams.md`

当修改 `.go` 文件时，必须优先遵守 Go 后端规范。
当修改 `.tsx`、`.ts`、`.jsx` 文件时，必须优先遵守 React 前端规范。
当修改 migration、SQL、schema 相关文件时，必须优先遵守 PostgreSQL 规范。
当使用多个 Agent 协作时，必须优先遵守 Agent Teams 协作规范，并读取 `.claude/agents` 中对应角色提示词。

## 通用工作原则

Agent 修改项目时必须先理解现有项目，再进行实现：

1. 先阅读项目目录、README、配置文件、构建脚本和与需求相关的代码。
2. 优先沿用项目已有技术栈、目录结构、命名方式、UI 风格和工具链。
3. 控制改动范围，只修改与当前需求直接相关的文件，避免顺手重构无关模块。
4. 新增依赖、改变目录结构、调整公共约定前，必须说明理由、影响范围和替代方案。
5. 修改完成后运行项目已有的格式化、类型检查、测试或构建命令；如无法运行，必须说明原因。
6. 交付说明必须包含修改范围、验证结果、未验证项和潜在风险。

## 个人开发者默认策略

本规范优先服务个人开发者长期维护项目，默认采用“够严谨，但不过度工程化”的策略。

- 小型工具、Demo、验证性项目：允许裁剪复杂工程要求，但必须保证可读、可运行、可继续维护。
- 长期维护项目：必须补齐必要的测试、README、错误处理、配置说明和工程化检查。
- SaaS、后台、支付、权限、数据迁移等高风险项目：必须严格执行对应技术规范中的安全、测试、审计和迁移要求。
- 不为了满足规范而引入不必要的框架、组件库、中间件或复杂抽象。
- 当规范与已有项目实践冲突时，优先保持已有项目一致性，并在交付说明中说明取舍。

## 模型适配原则

本项目可能使用 MiniMax-M2.7 或其他非 Claude 官方模型运行 Claude Code。为提升不同模型下的稳定性，规范必须保持显式、分步、可验证。

- 复杂任务必须先拆分步骤，再逐步执行。
- Agent 不得依赖隐式记忆，必须以当前仓库文件和用户本次要求为准。
- 修改前先读取相关文件，修改后必须检查 diff 或 Git 状态。
- 跨多个文件的修改必须说明改动边界，避免顺手扩大范围。
- 能运行验证命令时必须运行；不能运行时必须说明原因。
- 交付说明必须包含修改范围、验证结果、风险和后续建议。
- 遇到规则冲突时必须显式说明取舍，不得静默选择。
- 不确定项目约定时，优先读取 README、配置文件、脚本和相邻实现，而不是凭经验猜测。

## 项目计划书优先级

如果项目存在 `docs/project-plan.md`，Agent 在开发前必须先阅读该文件。

- 项目计划书定义“做什么”和“不做什么”。
- `.claude/rules` 定义“怎么做”。
- 当计划书与公共规范冲突时，优先遵守安全底线，其次遵守项目计划书，并在交付说明中说明取舍。
- 不得实现计划书明确写入“暂不实现”的功能。
- 如果用户需求超出计划书范围，必须先说明影响，再执行。
- 开始新项目开发时，建议先使用 `.claude/commands/start-project.md` 对计划书进行阶段拆分和风险检查。

## Agent Teams 协作策略

本项目支持在复杂需求中使用多个 Agent 协作，但默认不启用多 Agent。只有当需求跨前端、后端、数据库、测试、文档，或涉及权限、支付、迁移、隐私、安全等高风险场景时，才建议启用 Agent Teams。

启用前必须先阅读：

1. `.claude/rules/agent-teams.md`
2. `.claude/agents/lead-agent.md`
3. 本次需要启用的专业 Agent 角色文件
4. `docs/project-plan.md`，如果存在

默认由 `Lead Agent` 统一拆分任务、分配文件所有权、控制串并行顺序和最终验收。专业 Agent 只在自己的职责和文件边界内工作。

预设 Agent：

- `Lead Agent`：统筹、拆分、冲突处理、最终验收。
- `Product Planner Agent`：需求范围、阶段计划、验收标准。
- `Go Backend Agent`：Go 后端接口、业务逻辑、测试。
- `Frontend Agent`：Web 用户端、Web 后台管理端、H5 移动端和通用 React 前端开发。
- `Database Agent`：PostgreSQL 建模、迁移、约束、索引。
- `QA Review Agent`：代码审查、测试缺口、风险检查。
- `Docs Agent`：README、接口说明、使用和部署文档。

推荐使用 `.claude/commands/start-agent-team.md` 启动多 Agent 协作规划。用户确认团队组合、文件所有权和执行顺序前，不得直接大规模改代码。

## Git 与提交规则

- 默认不自动提交 Git。
- 只有用户明确要求提交，或项目文档明确授权自动提交时，Agent 才能执行 `git commit`。
- 提交前必须检查 `git status`，只提交与当前需求相关的文件。
- 未经用户授权时，只在交付说明中给出建议提交信息。
- 禁止把临时文件、构建产物、调试日志、无关改动和敏感配置混入提交。

## 规范冲突处理

- 安全、数据正确性、迁移安全和用户隐私要求优先级最高。
- 具体技术规范优先于本文件中的通用建议。
- 已有项目约定优先于新项目默认模板。
- 用户本次明确要求优先于默认建议，但不能覆盖安全底线。
